# 辅食助手产品架构设计文档

## 一、产品概述

### 1.1 产品定位
为2岁以下宝宝的宝妈提供个性化辅食食谱规划助手，通过对话式交互收集宝宝信息，生成一周7天的详细食谱（含烹饪步骤和配图），并支持后续调整。

### 1.2 核心功能
- **信息收集**：收集宝宝关键信息，判断营养需求
- **食谱生成**：生成一周7天完整食谱（名称、烹饪步骤、配图）
- **对话调整**：根据宝妈反馈调整食谱（口味、食材偏好）

## 二、需要收集的宝宝关键信息

### 2.1 基础信息
1. **宝宝年龄**（月龄，精确到月）
   - 用于判断辅食阶段（4-6月/7-9月/10-12月/13-24月）
   - 影响食材选择、质地、份量

2. **当前体重**（kg，可选）
   - 用于营养需求计算

3. **当前身高**（cm，可选）
   - 用于营养需求计算

### 2.2 发育阶段信息
4. **辅食添加时间**（已添加几个月）
   - 判断宝宝对辅食的适应程度

5. **当前进食能力**
   - 选项：纯母乳/配方奶 + 刚开始添加 / 已适应泥状 / 可吃软块状 / 可吃手指食物
   - 影响食谱质地和食材选择

6. **出牙情况**
   - 选项：未出牙 / 已出2-4颗 / 已出4-8颗 / 已出8颗以上
   - 影响食材硬度

### 2.3 健康与过敏信息
7. **已知过敏源**（多选）
   - 常见：鸡蛋、牛奶、花生、坚果、海鲜、大豆、小麦等
   - 必须严格规避

8. **特殊饮食需求**
   - 选项：无 / 素食 / 无麸质 / 其他（可填写）

9. **消化情况**
   - 选项：正常 / 容易便秘 / 容易腹泻 / 其他（可填写）
   - 影响食材选择（如便秘倾向多选高纤维）

### 2.4 偏好信息
10. **宝宝喜欢的食材**（可选，多选）
    - 常见食材列表：胡萝卜、南瓜、苹果、香蕉、鸡肉、鱼肉等
    - 优先在食谱中使用

11. **宝宝不喜欢的食材**（可选，多选）
    - 避免或减少使用

12. **家庭饮食习惯**
    - 选项：中式 / 西式 / 混合 / 其他
    - 影响食谱风格

### 3.2 技术栈选型

#### 前端
- **框架**：Next.js 14 (React) + TypeScript
- **UI组件库**：Ant Design / shadcn/ui
- **状态管理**：Zustand / React Query
- **实时通信**：WebSocket (Socket.io) 用于流式输出
- **样式**：Tailwind CSS

#### 后端
- **框架**：FastAPI (Python)
- **异步处理**：Celery + Redis
- **ORM**：SQLAlchemy
- **API文档**：自动生成 (Swagger)

#### 模型服务
- **LLM**：OpenAI GPT-4 / 通义千问 / 本地部署模型
- **图像生成**：DALL-E 3 / Midjourney API / Stable Diffusion
- **向量检索**：Chroma / Qdrant / Milvus
- **Embedding**：text-embedding-3-small / 中文embedding模型

#### 数据存储
- **关系数据库**：PostgreSQL (用户、宝宝信息、食谱、历史)
- **向量数据库**：Chroma / Qdrant (辅食知识库)
- **对象存储**：MinIO / 阿里云OSS / AWS S3 (图片存储)
- **缓存**：Redis (会话、临时数据)

## 四、核心业务流程

### 4.1 信息收集流程

```
用户进入 → 引导式表单/对话收集信息
  ↓
验证信息完整性
  ↓
调用营养分析引擎
  ↓
返回营养需求分析（展示给用户确认）
  ↓
用户确认 → 进入食谱生成
```

### 4.2 食谱生成流程

```
接收宝宝信息 + 营养需求
  ↓
检索知识库（辅食搭配、禁忌、营养数据）
  ↓
调用LLM生成一周食谱（结构化JSON）
  ↓
并行处理：
  ├─ 验证食谱合理性（营养均衡、过敏规避）
  ├─ 生成每日配图（调用图像生成API）
  └─ 生成详细烹饪步骤
  ↓
组装完整食谱数据
  ↓
存储到数据库
  ↓
返回给前端展示
```

### 4.3 对话调整流程

```
用户反馈（"宝宝不喜欢胡萝卜"）
  ↓
解析用户意图（食材替换/口味调整/份量调整）
  ↓
检索当前食谱
  ↓
调用LLM生成调整方案
  ↓
验证调整后食谱（营养、过敏、合理性）
  ↓
生成新配图（如食材变化）
  ↓
更新食谱数据
  ↓
返回调整后的食谱
```

- created_at, updated_at

## 七、模型策略设计

### 7.1 营养分析策略

**输入**：宝宝信息（年龄、体重、身高、发育阶段）

**处理逻辑**：
1. 根据年龄确定辅食阶段
2. 计算基础营养需求（参考WHO/中国营养学会标准）
3. 根据发育阶段调整（如出牙情况影响质地）
4. 根据过敏信息过滤食材

**输出**：结构化营养需求JSON

### 7.2 食谱生成策略

**Prompt设计**：
```
你是一位专业的婴幼儿营养师，需要为{age}个月的宝宝制定一周辅食食谱。

宝宝信息：
- 年龄：{age}个月
- 发育阶段：{feeding_stage}
- 过敏源：{allergies}
- 偏好：{preferences}
- 营养需求：{nutrition_requirements}

要求：
1. 生成7天完整食谱（早餐、午餐、晚餐、加餐）
2. 每道菜包含：名称、食材清单、详细烹饪步骤、营养说明
3. 确保营养均衡，规避过敏源
4. 符合宝宝当前发育阶段的食材质地
5. 输出JSON格式
```

**后处理**：
- 验证JSON格式
- 检查过敏规避
- 计算营养总和
- 生成配图提示词

### 7.3 配图生成策略

**方案1（推荐）**：使用图像生成API
- 提示词：`"精美的婴幼儿辅食照片，{dish_name}，{ingredients}，温馨的摆盘，专业摄影风格"`
- 模型：DALL-E 3 / Midjourney

**方案2（降级）**：使用图片库
- 建立辅食图片库，根据食材匹配相似图片

**方案3（成本优化）**：延迟生成
- 首先生成食谱，用户查看时再生成配图（异步）

### 7.4 对话调整策略

**意图识别**：
- 食材替换："不喜欢胡萝卜" → 识别胡萝卜，推荐替代食材
- 口味调整："太淡了" → 增加天然调味（如南瓜、苹果增加甜味）
- 份量调整："吃不完" → 减少份量

**调整流程**：
1. 解析用户意图（NER + 意图分类）
2. 检索当前食谱
3. 调用LLM生成调整方案
4. 验证调整后合理性
5. 如食材变化，重新生成配图

## 八、知识库设计

### 8.1 知识库内容
- **食材营养数据**：每100g的营养成分
- **辅食搭配规则**：哪些食材可以搭配，哪些不能
- **过敏源信息**：常见过敏源及替代方案
- **烹饪技巧**：不同月龄的烹饪方法
- **营养标准**：各月龄的营养需求参考

### 8.2 RAG检索流程
1. 用户查询 → Embedding向量化
2. 向量检索知识库（Top-K相似内容）
3. 将检索内容作为上下文注入Prompt
4. LLM基于知识库内容生成答案

## 九、前端页面设计

### 9.1 信息收集页 (`/onboarding`)
- **步骤1**：基础信息（年龄、体重、身高）
- **步骤2**：发育阶段（进食能力、出牙情况）
- **步骤3**：健康信息（过敏、消化情况）
- **步骤4**：偏好信息（喜欢/不喜欢的食材）
- **步骤5**：营养需求确认（展示分析结果）

### 9.2 食谱展示页 (`/recipes/:id`)
- **顶部**：一周日期导航（周一-周日）
- **左侧**：每日时间轴（早餐/午餐/晚餐/加餐）
- **右侧**：选中餐次的详细信息
  - 菜品名称
  - 配图（大图展示）
  - 食材清单
  - 详细烹饪步骤（分步展示）
  - 营养信息（卡路里、蛋白质等）

### 9.3 对话调整页 (`/recipes/:id/chat`)
- **左侧**：当前食谱预览（可折叠）
- **右侧**：对话界面
  - 消息列表（用户/助手）
  - 输入框（支持语音输入）
  - 快捷操作按钮（"替换食材"、"调整口味"等）

### 9.4 历史记录页 (`/history`)
- 列表展示所有历史食谱
- 支持按时间筛选
- 点击进入详情

## 十二、风险与应对

### 12.1 模型成本
- **风险**：图像生成API成本高
- **应对**：缓存机制、延迟生成、图片库降级

### 12.2 内容准确性
- **风险**：LLM可能生成不准确的营养信息
- **应对**：知识库约束、后处理验证、免责声明

### 12.3 用户体验
- **风险**：生成速度慢（特别是配图）
- **应对**：流式输出、异步生成、加载状态优化

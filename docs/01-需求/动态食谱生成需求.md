# 动态食谱生成需求

---

## 需求目标

将当前**静态内置默认食谱**升级为**火山引擎大模型实时动态生成**，实现：

1. **每次生成全新内容**：用户每次点击"生成食谱"都会得到一套全新的7天食谱，内容每天都不一样
2. **提升食谱多样性**：避免现有静态食谱重复性高的问题，给用户持续新鲜的体验
3. **换一道菜功能动态化**：原有的"换一道菜"功能也使用火山引擎模型即时生成，不再依赖静态替换库
4. **持久化积累**：所有生成的食谱保存到数据库，随着使用数据越来越丰富，弱网/无网时可以从数据库随机取用，保证多样性

---

## 功能点列表

### 1. 整体流程改造：动态生成7天食谱

| 项 | 说明 |
|---|------|
| 当前 | 使用内置静态代码，每次返回固定内容 |
| 改造后 | 调用火山引擎大模型，根据宝宝信息实时生成，每次内容都不一样 |

**需求细节**：
- 用户点击"生成食谱"后，向后端发送生成请求
- 后端根据宝宝信息（月龄、进食能力、过敏源、偏好等）构造prompt，调用火山引擎API
- 大模型直接生成完整7天 × 4餐 = 28道菜，保持原有JSON格式
- 生成结果保存到数据库后返回给前端
- 前端交互流程不变（流式输出逻辑保持不变，只是数据源从静态变动态）

---

### 2. "换一道菜"功能改造：动态生成替换菜品

| 项 | 说明 |
|---|------|
| 当前 | 使用静态分类替换库，从预定义列表随机选一道返回 |
| 改造后 | 调用火山引擎大模型，根据原菜品实时生成一道营养相似但食材不同的新菜品 |

**需求细节**：
- 用户点击某道菜的"换一道"按钮
- 后端将原菜品信息、宝宝信息、营养要求一起发给大模型
- 要求大模型生成：**同一餐次、相似营养成分、不同食材**的新菜品
- 新菜品直接替换数据库中原菜品记录
- 返回更新后的完整食谱给前端

---

### 3. 数据持久化与缓存策略

**设计思路**：
- 每次动态生成的食谱完整保存到数据库（已有 `recipes` + `recipe_details` 表结构，可直接复用）
- 随着用户使用，数据库中会积累越来越多不同的食谱菜品
- **降级策略**：当火山引擎API不可用（弱网/无网/限流），自动降级为从数据库随机抽取菜品组合
  - 从数据库中按营养类型分类随机抽取
  - 保证组合出来的食谱依然多样性，不会重复

**具体要求**：
- 新增 `GeneratedDish` 表（可选）或直接利用现有 `RecipeDetail` 表积累菜品数据
- 抽取出一个公共服务 `DishCacheService`，提供：
  - `save_dish()`：保存生成好的菜品
  - `random_sample_dish()`：按营养类型随机采样一道菜品
- 生成流程优先尝试调用API，失败自动降级到随机采样

---

### 4. 缓存降级机制

**降级触发条件**：
- 网络超时
- API返回错误
- API限流
- 无网络连接

**降级处理逻辑**：
1. 生成完整7天食谱时API失败 → 从数据库菜品库中随机组合出28道菜
2. 换一道菜时API失败 → 从对应营养类型中随机选一道不同的菜品

---

### 5. 配置项

需要在后端添加环境变量配置：
- `VOLCENGINE_API_KEY`：火山引擎API密钥
- `VOLCENGINE_BASE_URL`：火山引擎API地址（默认：https://aquasearch.volcengineapi.com）
- `VOLCENGINE_MODEL_NAME`：使用的模型名称（可配置）

## 验收标准

### 功能验收
- [ ] 用户每次点击"生成食谱"，得到的7天食谱**内容都不相同**
- [ ] 生成的食谱严格符合宝宝月龄、进食能力、过敏源等约束条件
- [ ] "换一道菜"功能点击后，能生成一道**营养相似但食材不同**的新菜品
- [ ] 所有生成的食谱和菜品都**持久化保存**在数据库中
- [ ] 当火山引擎API不可用时，系统**自动降级**从数据库取数据，依然能正常生成食谱，且保持多样性
- [ ] API Key `c27412f1-239d-453f-a0b9-5e5bd296b027` 正确配置使用

### 多样性验收
- [ ] 连续生成3次7天食谱，三次之间菜品重复率低于30%
- [ ] 同一道菜连续换3次，每次得到不同的菜品

### 数据持久化验收
- [ ] 每次生成的食谱都能在数据库 `recipes` 和 `recipe_details` 表中查到完整记录
- [ ] 积累的菜品数据可用于降级采样

### 降级机制验收
- [ ] 模拟API失败，系统能正常返回食谱，使用数据库随机组合
- [ ] 降级返回的食谱依然满足营养和过敏源约束

## 集成思路

### 火山引擎API调用流程
```
1. 构造prompt（宝宝信息 + 营养需求 + 格式要求）
2. 调用火山引擎聊天补全API
3. 解析返回的JSON
4. 保存到数据库
5. 返回给前端
```

### Prompt工程
- 7天食谱生成prompt复用现有结构，调整为给大模型的指令
- 换一道菜prompt需要明确："保持相同餐次，相似营养，不同食材"

### 错误处理
- API超时 → 降级
- API限流 → 降级
- JSON解析错误 → 降级
- 网络错误 → 降级

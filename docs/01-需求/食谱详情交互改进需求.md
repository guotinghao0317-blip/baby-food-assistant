# 食谱详情交互改进需求

## 1. 原始问题描述

用户合并分步生成页面到食谱详情页后，反馈交互细节问题：

1. **Tab聚合问题**：原来既有顶部进度卡片，底部又有Tab切换，重复了。需要聚合：周一到周日只用一组Tab组件，未生成置灰不可点击，生成完成后可点击，不需要分开两块。
2. **自动切换问题**：每生成新的一天，自动切换到那天，干扰用户当前浏览。要求：生成过程中不改变用户当前选中的Tab，保留用户选择。
3. **缓存+重新生成需求**：生成完成后缓存到localStorage，刷新页面不需要重新生成，提升体验。同时顶部增加「重新生成」按钮，点击可以清除缓存，重新生成一份完全不同的食谱，保证多样性。

## 2. 需求澄清过程

### 第一次理解（错误）：
- 理解为：保留顶部进度卡片，删除底部Tab
- 用户纠正：不要进度卡片，要用**Tab组件**，在Tab上显示进度状态

### 最终确认：
- 使用**一组Tab组件**展示周一到周日，不再分开两块
- 每个Tab通过样式/icon表达进度状态：
  - 未生成：文字置灰 + 不可点击
  - 生成中：正常颜色 + 旋转加载动画
  - 生成完成：正常颜色 + 对勾icon + 可点击
  - 选中：粉色背景区分选中态

## 3. 最终确认的改进方案

### 具体改进点

1. **Tab状态处理**：
   - 未生成：`bg-gray-50 text-gray-400 opacity-60 cursor-not-allowed`
   - 生成中：`text-pink-500` + `animate-spin` 旋转图标
   - 生成完成：`text-gray-700 hover:bg-gray-50 cursor-pointer` + ✓ 对勾
   - 选中：`bg-pink-500 text-white`

2. **保持用户选中状态**：
   - 生成新一天开始时，**不修改** `selectedDay`，保持用户当前选择
   - 只更新天的进度状态，不干扰用户操作

3. **localStorage缓存**：
   - 页面加载时：尝试从缓存读取已有生成结果
   - 如果缓存完整，直接恢复状态，不重新生成
   - 如果缓存部分完成，恢复进度继续生成剩余天数
   - 每次day_done自动保存进度到缓存

4. **「重新生成」按钮**：
   - 生成完成后在顶部标题栏显示
   - 点击后：清除缓存 → 重置所有状态 → 从头开始重新生成
   - 后端生成新recipe_id，天然保证全新内容，多样性满足

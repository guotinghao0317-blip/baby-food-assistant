# 食谱生成交互问题分析 - 技术设计文档

## 一、问题分析

### 问题1：交互表现 - 需要每道菜流式生成

**现状分析：**
- 后端 `generate_weekly_recipe_step_by_step` 函数生成每一天后，发送 `day_done` 事件
- `day_done` 事件包含完整的天数据（4道菜）
- 前端 `generate/page.tsx` 接收到 `day_done` 事件后，只是打印日志
- 用户需要等7天全部生成完成后，跳转到详情页才能看到内容

**根本原因：**
1. 前端 `generate/page.tsx` 只是一个进度显示页面，没有设计菜品展示区域
2. `day_done` 事件中的 `details` 数组被前端忽略了（第221-223行）
3. 当前流程是：generate页面显示进度 -> 完成后跳转到 `[id]/page.tsx` 详情页查看

**预期行为：**
- 生成过程中，每道菜生成后立即显示（打字机效果）
- 用户在generate页面就能看到生成的菜品，而不是只看到进度条

---

### 问题2：页面加载问题

**现状分析：**
- 用户从 generate 页面点击"重新生成"后，清除缓存并重置状态
- 重新生成完成后，用户仍停留在 generate 页面
- 需要手动切换到 `[id]/page.tsx` 详情页才能看到最新数据

**根本原因：**
1. generate 页面和详情页是两个独立的页面
2. generate 页面设计初衷是"生成入口"，不是查看页面
3. 跳转逻辑只在首次生成时执行（第236-239行的 `setTimeout` 跳转）

**预期行为：**
- 7天全部生成完成后，前端自动跳转到详情页
- 用户进入详情页时能看到完整的食谱，不需要手动操作

---

### 问题3：数据去重问题

**现状分析：**
- 后端在保存数据时会检查重复（recipe_generator.py 第618-627行）
- 但检查的是 `dish_name` 重复，而不是 `meal_type` 重复
- 数据库中可能存在同一天、同一 `meal_type` 的多道菜
- 前端显示时，使用 `organizedRecipes[day_of_week][meal_type]` 的映射方式，导致重复的 `meal_type` 只会显示最后一道

**根本原因：**
1. 后端 `generate_day_meals` 算法可能生成同一天同一 `meal_type` 的多道菜
2. 保存时的重复检查只检查菜品名称，不检查餐次
3. 前端映射方式 `organizedRecipes[day_of_week][meal_type] = detail` 会覆盖重复的 `meal_type`

**预期行为：**
- 每餐只有一道菜
- 后端生成时避免生成重复的 `meal_type`
- 数据库中同一天同一 `meal_type` 只保留一道菜

---

## 二、整体技术方案

### 方案概述

针对三个问题，采用以下综合解决方案：

1. **问题1方案：** 修改generate页面，增加菜品实时展示区域，将SSE事件中的菜品数据实时渲染
2. **问题2方案：** 优化页面跳转逻辑，生成完成后自动跳转到详情页，确保数据已加载
3. **问题3方案：** 在后端生成算法和保存逻辑中增加 `meal_type` 去重检查

### 架构调整

```
现状：
generate/page.tsx (只显示进度) -> 跳转 -> [id]/page.tsx (显示详情)

优化后：
generate/page.tsx (进度+实时菜品展示) -> 自动跳转 -> [id]/page.tsx (完整数据+去重)
```

### 模块交互流程

```
用户点击生成
    ↓
调用 POST /api/recipes/generate-stream
    ↓
后端创建recipe记录，status=generating
    ↓
循环7天生成：
    ├─ 发送 day_started 事件
    ├─ 生成4道菜（检查meal_type去重）
    ├─ 保存到数据库（检查重复）
    └─ 发送 day_done 事件（包含4道菜数据）
        ↓
    前端实时渲染菜品到页面
    ↓
发送 finished 事件
    ↓
前端自动跳转到 /recipes/[id] 详情页
```

## 七、风险与注意事项

### 风险1：SSE 事件处理时序
**风险：** 如果 SSE 事件到达速度过快，前端状态更新可能出现竞态条件

**缓解措施：**
- 使用 React 的 `setState` 函数式更新，确保状态基于最新值
- 增加 `finished` 事件延迟，给前端充足时间处理

### 风险2：内存占用
**风险：** generate 页面缓存 7 天 * 4 道菜 = 28 道菜的数据，可能占用较多内存

**缓解措施：**
- 菜品数据结构简洁，只包含必要字段
- 跳转到详情页后，generate 页面卸载，自动释放内存

### 风险3：meal_type 去重导致数据不足
**风险：** 如果去重过于严格，可能导致某天菜品数量不足 4 道

**缓解措施：**
- 去重只在保存环节检查，生成环节不受影响
- 如果出现数据不足，重新生成该天（当前逻辑已支持）

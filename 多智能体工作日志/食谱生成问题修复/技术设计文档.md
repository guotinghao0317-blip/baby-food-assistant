# 食谱生成问题修复 - 技术设计文档

## 一、问题分析

### 1.1 用户反馈的问题

1. **食谱生成页显示为空**
   - 用户访问 `/recipes/generate?baby_id=xxx` 页面后，食谱生成页显示为空
   - 可能是SSE流式连接有问题，或者数据没有正确返回

2. **一周的食谱没办法一起生成（周三-周日的tab无法点击）**
   - 在食谱详情页 `/recipes/[id]/page.tsx` 中，周三到周日的标签页无法点击
   - 用户只能看到周一和周二的食谱，无法查看周三到周日的内容

### 1.2 根本原因分析

#### 问题1：食谱生成页显示为空

**分析过程：**
1. 查看前端 `generate/page.tsx`，发现它使用的是 `/api/recipes/generate-stream` 端点
2. 前端代码通过 `parseSSE()` 函数解析服务器发送事件
3. 前端期望的事件类型：
   - `started`: 开始生成
   - `day_started`: 开始生成某一天
   - `day_done`: 某一天生成完成
   - `chunk`: 内容更新
   - `finished`: 生成完成
   - `error`: 错误

**后端分析：**
1. 查看后端 `recipes.py` 的 `generate_recipe_stream` 端点（第217-273行）
2. 该端点调用 `generate_weekly_recipe_step_by_step` 函数
3. 该函数在 `recipe_generator.py` 中定义（第521-641行）
4. **关键发现：** 后端正确实现了 SSE 流式输出，但可能存在以下问题：
   - 前端和后端的事件格式不匹配
   - 连接超时或中断
   - 数据没有及时保存到数据库
   - 前端跳转逻辑有问题

**具体问题定位：**
- 前端代码第134行：调用 `/api/recipes/generate-stream`
- 后端路由第217行：定义了 `/generate-stream` 端点
- 前端期望收到 `finished` 事件后跳转到 `/recipes/${data.recipe_id}`
- 后端确实发送了 `finished` 事件（第636行）
- **问题：** 前端可能在事件处理过程中没有正确更新 UI 状态

#### 问题2：周三-周日Tab无法点击

**分析过程：**
1. 查看前端 `[id]/page.tsx`（第416-422行）的 `handleDayClick` 函数
2. 该函数的条件判断：`if (day.status === 'done' || day.status === 'generating')`
3. 只有当天的状态是 `done` 或 `generating` 时才能点击

**后端分析：**
1. 前端在 `fetchRecipe()` 函数中（第174-226行）尝试从数据库加载已有数据
2. 它会检查 `recipeData.details` 并更新 `days` 状态
3. **关键发现：** 前端期望从数据库返回的 `details` 中提取 `day_of_week` 信息

**具体问题定位：**
- 前端代码第196-216行：尝试从 `recipeData.details` 中提取已生成的天数
- 但是如果后端返回的数据格式不正确，或者某些天的数据缺失，那么这些天将保持 `pending` 状态
- `pending` 状态的 Tab 会被置灰且不可点击（第782-784行）
- **问题：** 可能是数据没有正确保存到数据库，或者查询返回的数据不完整

### 1.3 数据流分析

#### 食谱生成流程

```
用户访问 /recipes/generate?baby_id=xxx
    ↓
前端检查 token 和 baby_id
    ↓
前端调用 POST /api/recipes/generate-stream
    ↓
后端创建 Recipe 记录（status='generating'）
    ↓
后端调用 generate_weekly_recipe_step_by_step()
    ↓
循环7天（day 1-7）：
  - 发送 'day_started' 事件
  - 生成当天4道菜
  - 保存到数据库（RecipeDetail）
  - 发送 'day_done' 事件
    ↓
更新 Recipe.status = 'completed'
    ↓
发送 'finished' 事件
    ↓
前端收到 'finished' 事件
    ↓
前端跳转到 /recipes/{recipe_id}
```

#### 食谱详情页流程

```
用户访问 /recipes/{id}
    ↓
前端初始化 days 状态（7个pending）
    ↓
前端调用 GET /api/recipes/{id}
    ↓
后端查询 Recipe 和 RecipeDetail
    ↓
前端根据返回的 data.details 更新 days 状态
    ↓
前端开始自动生成（startGeneration）
    ↓
轮询检查状态并生成下一天
    ↓
用户可以点击已完成的 Tab 查看内容
```

## 二、技术方案设计

### 2.1 整体架构调整

本次修复不需要调整整体架构，主要是修复现有功能中的问题。

### 2.2 问题1修复方案：食谱生成页显示为空

#### 方案A：前端状态管理优化
- **问题：** 前端在处理 SSE 事件时可能没有正确更新 loading 状态
- **解决：**
  1. 在 `started` 事件处理中确保 `loading=true`
  2. 在 `finished` 事件处理中确保 `loading=false`
  3. 添加更多日志输出来调试问题

#### 方案B：后端事件格式验证
- **问题：** 后端发送的事件格式可能与前端期望的不一致
- **解决：**
  1. 验证 `generate_weekly_recipe_step_by_step` 发送的所有事件格式
  2. 确保所有事件都有正确的 `type` 字段
  3. 添加错误处理和日志

#### 方案C：连接稳定性优化
- **问题：** SSE 连接可能因为超时或网络问题中断
- **解决：**
  1. 前端添加重连逻辑
  2. 后端添加心跳机制
  3. 增加超时时间

#### 推荐方案：综合方案A + 方案B

### 2.3 问题2修复方案：周三-周日Tab无法点击

#### 方案A：数据完整性检查
- **问题：** 数据库中可能只有周一、周二的数据
- **解决：**
  1. 在前端加载后检查 `details.length` 是否等于 28（7天×4餐）
  2. 如果数据不完整，自动触发缺失天数的生成
  3. 添加调试日志显示已加载的天数

#### 方案B：前端状态初始化优化
- **问题：** 前端初始化 `days` 状态时可能没有正确同步数据库状态
- **解决：**
  1. 在 `fetchRecipe()` 完成后，立即更新 `days` 状态
  2. 确保所有已生成的天都被标记为 `done`
  3. 添加验证逻辑：如果有数据但状态还是 pending，强制更新为 done

#### 方案C：后端数据返回优化
- **问题：** 后端查询可能没有返回所有数据
- **解决：**
  1. 检查 `/api/recipes/{id}` 端点是否返回完整的 `details` 数组
  2. 确保数据库查询没有遗漏
  3. 添加数据验证逻辑

#### 推荐方案：综合方案A + 方案B

### 2.4 模块交互设计

```
前端 generate/page.tsx
    ↓ (SSE)
后端 /api/recipes/generate-stream
    ↓
generate_weekly_recipe_step_by_step()
    ↓ (保存数据)
数据库 Recipe + RecipeDetail
    ↓
前端跳转到 /recipes/{id}
    ↓
前端 [id]/page.tsx
    ↓ (REST)
后端 /api/recipes/{id}
    ↓
前端显示食谱并自动生成剩余天数
```

## 四、接口定义

### 4.1 现有接口（无需修改）

#### POST /api/recipes/generate-stream
**请求：**
```json
{
  "baby_id": 1
}
```

**响应（SSE流式）：**
```
data: {"type": "started", "recipe_id": 123}

data: {"type": "day_started", "day": 1}

data: {"type": "day_done", "day": 1, "details": [...], "generation_source": "codingplan"}

data: {"type": "day_started", "day": 2}

data: {"type": "day_done", "day": 2, "details": [...], "generation_source": "codingplan"}

...

data: {"type": "finished", "recipe_id": 123, "total_dishes": 28}
```

{

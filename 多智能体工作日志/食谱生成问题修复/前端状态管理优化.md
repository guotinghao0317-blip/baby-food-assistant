# 食谱生成页 - 前端状态管理优化

## 问题描述
用户访问 `/recipes/generate?baby_id=xxx` 后，食谱生成页显示为空（显示骨架屏），没有正确进入 loading 状态。

## 问题分析

### 发现的问题

1. **重复执行问题**
   - `useEffect` 可能在 React Strict Mode 下被调用两次
   - 没有 guard 防止重复执行

2. **状态初始化问题**
   - 页面加载时 `loading` 状态为 `false`
   - 虽然 `generateRecipe` 会设置 `loading=true`，但存在时序问题

3. **调试信息不足**
   - 关键状态转换缺少日志输出
   - 难以追踪问题

4. **错误处理不完善**
   - 错误发生后部分状态未重置
   - HTTP 错误响应体未读取

## 解决方案

### 1. 添加防重复执行机制

```typescript
const hasStartedRef = useRef(false)

useEffect(() => {
  // 避免重复执行
  if (hasStartedRef.current) {
    console.log('已开始生成，跳过重复执行')
    return
  }
  // ... 其他逻辑
  hasStartedRef.current = true
}, [searchParams, router])
```

### 2. 完善日志输出

在关键位置添加详细的日志：

- **useEffect 入口**：记录进入条件和参数
- **generateRecipe 入口**：记录状态重置
- **SSE 事件处理**：记录每个事件的类型和数据
- **错误处理**：记录错误类型和消息
- **状态更新**：记录关键状态变化

### 3. 优化错误处理

- 读取 HTTP 错误响应体
- 错误发生时重置所有相关状态
- 保持 loading 状态直到用户重试

### 4. 添加调试信息

在初始状态显示区域添加调试信息：

```typescript
<div className="mt-8 text-xs text-gray-400">
  初始状态: loading={String(loading)}, error={error}, currentStep={currentStep}, recipe={String(!!recipe)}
</div>
```

## 事件处理流程

### 后端 SSE 事件类型

| 事件类型 | 说明 | 前端处理 |
|---------|------|---------|
| `started` | 开始生成 | 设置 currentStep=1 |
| `day_started` | 开始生成某天 | 根据 day 设置对应步骤(1-7) |
| `day_done` | 某天生成完成 | 记录日志 |
| `chunk` | 内容更新 | 根据内容长度推断当前步骤 |
| `finished` | 生成完成 | 设置完成状态，800ms 后跳转 |
| `error` | 发生错误 | 抛出错误，显示错误信息 |

### 状态转换

```
初始状态 (loading=false, currentStep=0)
    ↓
[调用 generateRecipe]
    ↓
loading=true, currentStep=1
    ↓
[接收 started 事件]
    ↓
[接收 day_started 事件]
    ↓
currentStep 更新 (1-7)
    ↓
[接收 chunk 事件]
    ↓
根据内容长度推断步骤
    ↓
[接收 finished 事件]
    ↓
设置 recipe, 800ms 后跳转
```

## 核心改动思路

1. **防止重复执行**：使用 `useRef` 标记是否已开始，避免 React Strict Mode 导致的双次调用

2. **完善日志**：在关键位置添加带前缀的日志（`[useEffect]`、`[generateRecipe]`），便于快速定位问题

3. **优化错误处理**：读取完整的错误响应，重置所有相关状态，确保 UI 一致

4. **状态可视化**：在开发环境添加状态调试信息，便于快速发现问题

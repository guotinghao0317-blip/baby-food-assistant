# 食谱流式展示优化V2 - 技术设计文档

## 一、根因分析验证

### 问题1：算法路径瞬间完成，3道菜几乎同时出现

**验证结果：确认**

- 文件：`backend/app/services/recipe_generator.py` 第375-392行
- 算法路径调用 `generate_single_meal()` 是同步执行，瞬间完成
- 完成后仅发送1个 `dish_stream` 事件（整包推送），随后 `await asyncio.sleep(0.1)` 仅100ms延迟
- 紧接着就发送 `dish_done` 事件
- 同一天3道菜（breakfast/lunch/dinner）之间几乎无间隔，SSE事件到达前端的时间差在100-200ms内
- React 18 批量处理 `setDays`，导致3道菜在同一渲染帧内同时出现

### 问题2：day_done 事件后前端调用 fetchRecipe() 覆盖了流式状态

**验证结果：确认，且影响范围比预期更大**

- 文件：`frontend/app/recipes/[id]/page.tsx` 第588行
- `day_done` 处理器在第571-586行正确构建了 `dishes` Record，但紧接着第588行调用 `fetchRecipe()`
- `fetchRecipe()`（第212-281行）从后端 `/api/recipes/${recipeId}` 拉取完整数据，然后调用 `setDays()` 重写整个days数组
- **竞态条件**：`fetchRecipe()` 是异步的，在其响应到达前，后续的SSE事件（如 `dish_started`、`dish_stream`）可能已更新了其他天的状态；当 `fetchRecipe()` 响应到达时，会覆盖这些最新的流式状态
- **覆盖后果**：正在 generating 状态的天可能被回退为 pending（如果DB中尚未写入该天数据），或 dishes Record 被旧数据替换
- **同样的问题存在于**：`finished` 事件处理器（第636行）也调用了 `fetchRecipe()`，虽然在全部完成后调用影响较小，但仍然可能与最后的SSE事件产生竞态

### 问题3：前端从数据库加载已有食谱时，使用旧的 organizedRecipes 路径

**验证结果：部分确认，需补充细节**

- 文件：`frontend/app/recipes/[id]/page.tsx` 第1285-1310行
- 渲染分支优先级：先检查 `days[selectedDay-1]` 的 status + displayDishes，若不满足则 fallback 到 `canShowFromOrganizedRecipes`
- **`fetchRecipe()` 实际上已同时填充 `dishes` Record**（第260-267行），所以正常情况下加载已完成食谱后，新的 dishes 渲染路径会被优先使用
- **但存在边界情况**：
  1. `fetchRecipe()` 异步完成前，`days` 初始状态为 `pending`，此时若 `recipe` 也未加载，两个路径都不命中，用户看到"等待生成中..."
  2. 如果 `fetchRecipe()` 只设置了部分天（DB中只有部分天数据），未完成的天仍为 `pending`，此时 `displayDishes` 为空，fallback 到 `organizedRecipes` 也为空，同样显示空白
  3. `organizedRecipes` 渲染路径（第1289-1316行）的UI样式与 dishes 渲染路径不一致（不同的className、不同的卡片结构），导致视觉闪烁

### 补充发现

- `generate/page.tsx` 和 `generate-step-by-step/page.tsx` 不存在 Problem 2 和 Problem 3（没有 `fetchRecipe()` 和 `organizedRecipes`），仅受 Problem 1 影响
- `generate-step-by-step/page.tsx` 的 `day_done` 处理器（第254-273行）没有调用 `fetchRecipe()`，不存在覆盖问题

## 二、技术方案

### 任务1 [后端] 算法路径模拟流式延迟

**目标**：算法路径降级时，每道菜之间有明显的视觉间隔，模仿LLM流式输出的节奏感

**方案**：

1. 在算法路径的每道菜之间增加 `asyncio.sleep` 延迟
2. 将整包推送改为分块推送，模拟 CodingPlan API 的流式行为
3. 具体修改点（`recipe_generator.py`）：
   - 第388-392行：算法路径生成 `dish_data` 后，不立即发送完整 `dish_stream` 事件
   - 改为：先发送 `dish_stream` 事件标记为 streaming，等待 600-800ms，再发送 `dish_done` 事件
   - 在每道菜循环结束（`dish_done` 之后）增加 400-600ms 延迟，确保下一道菜的 `dish_started` 与上一道的 `dish_done` 之间有间隔
4. 延迟时间参考值（可配置为常量）：
   - `STREAMING_DISPLAY_DELAY = 0.8` — streaming 状态展示时间
   - `DISH_INTERVAL_DELAY = 0.5` — 菜与菜之间的间隔

**风险**：增加延迟会延长整体生成时间（算法路径3道菜增加约3-4秒），但这是必要的用户体验权衡

---

### 任务2 [前端] 移除 day_done 中的 fetchRecipe() 调用

**目标**：消除 `fetchRecipe()` 对流式状态的覆盖和竞态条件

**方案**：

1. **`[id]/page.tsx` 第588行**：删除 `day_done` 处理器中的 `fetchRecipe()` 调用
   - `day_done` 事件已包含完整的 `details` 数据，前端已在第571-586行将其转换为 `dishes` Record
   - 无需再从后端拉取重复数据
2. **`[id]/page.tsx` 第636行**：保留 `finished` 事件中的 `fetchRecipe()` 调用，但增加保护逻辑
   - 在 `fetchRecipe()` 的 `setDays()` 回调中，仅更新 status 为 `pending` 的天（避免覆盖 generating 状态的天，虽然此时理论上不应该有 generating 的天）
   - 或者更安全地：在 `finished` 事件到达后延迟 500ms 再调用 `fetchRecipe()`，确保所有 SSE 事件已处理完毕

**影响范围**：仅 `frontend/app/recipes/[id]/page.tsx`

---

### 任务3 [前端] 统一渲染路径，移除 organizedRecipes fallback

**目标**：消除两条渲染路径的视觉不一致，确保所有场景都走 `dishes` Record 渲染

**方案**：

1. **删除 `organizeRecipes()` 函数**（第885-898行）及其相关变量
2. **删除 `canShowFromOrganizedRecipes` 判断**（第1176行）
3. **修改渲染分支逻辑**（第1285行附近）：
   - 原：`status === 'generating' || status === 'done'` 时用 dishes 路径，否则 fallback 到 organizedRecipes
   - 新：只要 `displayDishes.length > 0` 就用 dishes 路径；如果 status 是 `pending` 且 `displayDishes` 为空，显示"等待生成中..."
   - 移除整个 organizedRecipes 渲染分支（第1289-1316行）
4. **确保 `fetchRecipe()` 加载的数据正确填充 `dishes`**：
   - 现有 `fetchRecipe()` 第260-267行已正确填充，无需修改
   - 但需要确认：从DB加载的 `detail` 数据结构是否与 `DishGenerationState.detail` 兼容（字段名可能不同，如 `day_of_week` vs `day`）
5. **处理重新生成单道菜的场景**：
   - 现有的 `organizedRecipes` 路径中有 `isRegenerating` 和 `streamingDish` 的逻辑
   - 需要将这部分逻辑迁移到 dishes 路径的 `renderDishCard` 中

**影响范围**：`frontend/app/recipes/[id]/page.tsx`

---

### 任务4 [前端] fetchRecipe() 安全保护

**目标**：确保 `fetchRecipe()` 不会覆盖正在生成的流式状态

**方案**：

1. **在 `fetchRecipe()` 的 `setDays()` 回调中增加保护条件**（第251行附近）：
   - 遍历 `prevDays`，只更新 status 为 `pending` 的天，保留 `generating` 和 `done` 天的现有状态
   - 或者：只更新 `fetchRecipe()` 返回数据中确实存在的天，且该天在前端不处于 `generating` 状态
2. **`fetchRecipe()` 返回后同步 `recipe` 状态**：
   - 确保 `setRecipe()` 和 `setDays()` 在同一批处理中更新，避免中间状态
3. **增加去重逻辑**：
   - 如果 `fetchRecipe()` 返回的某天数据与前端 `days` 中的数据一致（同一天、同菜品、同状态），跳过更新

**影响范围**：`frontend/app/recipes/[id]/page.tsx`

## 四、验收标准

### 场景1：算法路径降级时的逐菜展示

1. CodingPlan API 不可用时，算法路径生成的食谱应以逐菜方式展示
2. 每道菜之间应有 **至少400ms** 的视觉间隔，用户能清晰看到：pending -> generating -> streaming -> done 的状态转换
3. 同一天3道菜不会在同一帧内同时出现

### 场景2：day_done 事件后状态不被覆盖

1. 第1天 day_done 后，第2天的 generating/streaming 状态不受影响
2. 流式生成过程中，已完成的菜品卡片不会出现"消失"或"闪烁"
3. 全部7天完成后，`fetchRecipe()` 正常同步最终数据到 `recipe` 状态（用于重新生成单道菜等功能）

### 场景3：重新进入食谱详情页

1. 用户关闭页面后重新进入 `/recipes/[id]`，已完成的食谱正常显示
2. 所有天的菜品都通过 `dishes` Record 渲染路径展示，不出现 organizedRecipes 旧渲染路径
3. 视觉样式与生成过程中一致，无闪烁

### 场景4：generate/page.tsx 和 generate-step-by-step/page.tsx

1. 两个页面在算法路径降级时同样呈现逐菜展示效果
2. 现有功能不受影响（这两个页面不存在 Problem 2 和 Problem 3）

### 回归验收

1. CodingPlan API 正常可用时，流式生成行为不变
2. 重新生成单道菜功能正常工作
3. 缓存机制（loadCache/saveCache）正常工作
4. 三个生成页面（generate、generate-step-by-step、[id]）的核心流程不受影响

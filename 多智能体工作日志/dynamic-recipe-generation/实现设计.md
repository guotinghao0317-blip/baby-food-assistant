## 1. 需求分析与整体技术方案

### 1.1 需求理解

当前系统使用**静态内置默认食谱**，存在重复性高、多样性不足的问题。需要升级为**火山引擎大模型实时动态生成**，核心目标：

1. **每次生成全新内容**：用户每次点击"生成食谱"都得到全新7天食谱（28道菜）
2. **提升食谱多样性**：避免静态食谱重复性高的问题
3. **"换一道菜"动态化**：原有的替换功能也使用大模型即时生成
4. **持久化积累与降级**：所有生成结果保存到数据库，API不可用时自动降级从数据库随机采样

### 1.2 整体架构

```
┌─────────────┐
│   前端       │
└──────┬──────┘
       │ HTTP请求
       ▼
┌───────────────────────────────────┐
│  后端路由层 (recipes.py)           │
│  - 接口协议保持不变                 │
└───────────────┬───────────────────┘
                │
        ┌───────▼──────────┐
        │  生成/替换服务    │
        └───┬──────────────┘
            │
    ┌───────▼─────────────────┐  失败自动降级    ┌───────────────────┐
    │  火山引擎API客户端       │ ──────────────▶ │  菜品缓存服务      │
    │  (volcengine_client.py) │                 │  (dish_cache.py)  │
    └─────────────────────────┘                 └─────────┬─────────┘
                                                        │
                                                ┌───────▼──────────┐
                                                │   数据库         │
                                                │  RecipeDetail    │
                                                └──────────────────┘
```

### 1.3 技术选型

| 组件 | 技术方案 | 说明 |
|------|---------|------|
| API调用 | HTTP + JSON | 直接调用火山引擎开放API，使用 `httpx` |
| 认证 | API Key 在 Header 中 | 按火山引擎要求传递 |
| 持久化 | 复用现有 `RecipeDetail` 表 | 不需要新建表，简化设计 |
| 缓存降级 | 按营养类型随机采样 | 复用现有 `dish_replacer.py` 的分类逻辑 |
| 超时控制 | 30秒超时 | 超过后立即触发降级 |

## 3. 降级缓存机制设计

### 3.1 设计目标

当火山引擎API不可用（弱网、无网、限流）时，系统依然能正常工作，且保证**多样性**，不会因为用静态数据回到重复问题。

### 3.2 数据存储方案

**选择：复用现有 `RecipeDetail` 表，不需要新建表**。

理由：
- 每次生成的食谱已经完整保存到 `recipes` + `recipe_details`
- 所有动态生成的菜品天然积累在 `recipe_details` 中
- 不需要额外的数据迁移，表结构已经满足需求
- 随着使用，数据库中菜品数量会越来越多，多样性自然提升

### 3.3 菜品分类策略

复用 `dish_replacer.py` 已有的 `classify_dish_nutrition_type()` 分类逻辑：

| 分类Key | 说明 |
|---------|------|
| `iron_red_meat` | 红肉补铁类 |
| `protein_poultry` | 白肉/禽肉高蛋白类 |
| `seafood_fish` | 鱼虾水产类 |
| `porridge` | 粥类主食 |
| `vegetable_fruit` | 蔬果泥/蔬果块 |
| `iron_rice` | 高铁米糊类 |

这个分类足够支持按营养类型随机采样，保证替换/组合时营养均衡。

### 3.4 DishCacheService 设计

```python
# backend/app/services/dish_cache.py

from typing import Dict, List, Optional
import random
from sqlalchemy.orm import Session
from app.models import RecipeDetail

class DishCacheService:
    def __init__(self, db: Session):
        self.db = db
    
    def save_dish(self, recipe_id: int, dish_data: Dict) -> RecipeDetail:
        """保存生成好的菜品到数据库（已经由路由层保存，这里主要用于索引）"""
        # 实际上因为路由层已经保存到 RecipeDetail，这里不需要额外操作
        # 只需要确保数据被持久化即可
        pass
    
    def random_sample_dish(
        self,
        nutrition_type: str,
        exclude_dish_ids: List[int] = None
    ) -> Optional[Dict]:
        """
        按营养类型随机采样一道菜品
        - exclude_dish_ids: 排除已使用过的菜品ID，避免重复
        """
        # Query all dishes in this nutrition type category
        # 我们需要分类，所以在内存过滤（数据量不大，完全可接受）
        all_dishes = self.db.query(RecipeDetail).all()
        
        # 分类过滤
        from app.services.dish_replacer import classify_dish_nutrition_type
        candidates = [
            dish for dish in all_dishes
            if classify_dish_nutrition_type(dish) == nutrition_type
            and (exclude_dish_ids is None or dish.id not in exclude_dish_ids)
        ]
        
        if not candidates:
            return None
        
        # 随机选择一道
        selected = random.choice(candidates)
        return self._dish_to_dict(selected)
    
    def random_sample_dishes_by_meal(
        self,
        day_of_week: int,
        meal_type: str,
        exclude_ids: List[int]
    ) -> Dict:
        """按餐次随机采样一道（用于完整7天生成降级）"""
        # 先获取该餐次所有菜品
        all_dishes = self.db.query(RecipeDetail)\
            .filter(RecipeDetail.meal_type == meal_type)\
            .filter(~RecipeDetail.id.in_(exclude_ids))\
            .all()
        
        if not all_dishes:
            #  fallback 到按营养分类
            from app.services.dish_replacer import classify_dish_nutrition_type
            all_dishes = self.db.query(RecipeDetail).all()
            # 根据餐次猜大概类型
            if meal_type in ["breakfast"]:
                # 早餐多是粥/米糊
                candidates = [d for d in all_dishes 
                             if classify_dish_nutrition_type(d) in ["porridge", "iron_rice"]
                             and d.id not in exclude_ids]
            elif meal_type in ["lunch", "dinner"]:
                # 午餐晚餐多是主菜
                candidates = [d for d in all_dishes 
                             if classify_dish_nutrition_type(d) in ["iron_red_meat", "protein_poultry", "seafood_fish"]
                             and d.id not in exclude_ids]
            else:
                # 加餐多是蔬果
                candidates = [d for d in all_dishes 
                             if classify_dish_nutrition_type(d) == "vegetable_fruit"
                             and d.id not in exclude_ids]
        
            if not candidates:
                candidates = [d for d in all_dishes if d.id not in exclude_ids]
            
            if not candidates:
                candidates = all_dishes
        
            selected = random.choice(candidates)
        else:
            selected = random.choice(all_dishes)
        
        return self._dish_to_dict(selected)
    
    def get_statistics(self) -> Dict[str, int]:
        """获取缓存统计信息（用于监控）"""
        total = self.db.query(RecipeDetail).count()
        return {"total_dishes": total}
    
    def _dish_to_dict(self, dish: RecipeDetail) -> Dict:
        """转换为字典格式"""
        return {
            "day_of_week": dish.day_of_week,
            "meal_type": dish.meal_type,
            "dish_name": dish.dish_name,
            "ingredients": dish.ingredients,
            "cooking_steps": dish.cooking_steps,
            "nutrition_info": dish.nutrition_info,
            "id": dish.id
        }
```

### 3.5 降级触发流程

#### 完整7天生成降级：

```
1. 用户请求生成7天食谱
2. 尝试调用火山引擎API
   ├─ 成功 → 保存到数据库 → 返回
   └─ 失败 → 触发降级
       ↓
3. 从 DishCacheService 按天×餐次随机采样
   - 28道菜，每道菜都排除已经选过的ID，避免同一食谱内重复
   - 保持餐次不变（早餐还是早餐）
4. 组装完整7天食谱
5. 保存到数据库（降级生成的也保存，积累数据）
6. 返回给前端
```

#### 换一道菜降级：

```
1. 用户请求换一道菜
2. 尝试调用火山引擎API
   ├─ 成功 → 更新数据库 → 返回
   └─ 失败 → 触发降级
       ↓
3. 判断原菜品营养类型
4. 从对应分类中随机选一道不同的菜品
5. 替换原有记录
6. 返回给前端
```

### 3.6 多样性保证策略

1. **随机采样，排除已选**：同一食谱生成时，已经选过的菜品ID会被排除，避免一餐多吃重复
2. **分类约束营养均衡**：保持原餐次的营养类型，不会破坏营养均衡
3. **持续积累**：每一次生成（包括API成功和降级生成）都会保存到数据库，数据越来越多，采样空间越来越大，多样性持续提升
4. **初始数据足够**：即使还没有任何动态生成数据，系统已有静态菜品，降级依然可以工作

# 后端 meal_type 去重逻辑 - 设计文档

## 问题描述

用户反馈在食谱详情页看到菜品数量异常：

### 问题现象
控制台日志显示每天菜品数量如下：
```
第 1 天 (周一) 已完成，菜品数: 8  ← 期望4道，实际8道（有重复）
第 2 天 (周二) 已已完成的，菜品数: 3  ← 期望4道，实际3道（缺1道）
第 3 天 (周三) 已完成，菜品数: 8  ← 期望4道，实际8道（有重复）
第 4 天 (周四) 已完成，菜品数: 4  ← 正常
第 5 天 (周五) 已完成，菜品数: 8  ← 期望4道，实际8道（有重复）
第 6 天 (周六) 已完成，菜品数: 2  ← 期望4道，实际2道（缺2道）
第 7 天 (周日) 已完成，菜品数: 5  ← 期望4道，实际5道（多1道）
```

**关键发现：**
1. 有些天的菜品数量不是4道（每餐1道）
2. 数据库中可能存在同一天、同一meal_type的多道菜
3. 前端显示时只显示最后一个meal_type的菜品（因为被覆盖）

##根本原因分析

### 后端问题
1. `recipe_generator.py` 的 `generate_weekly_recipe_step_by_step` 函数在保存数据时，只检查了 `dish_name` 重复
2. **没有检查 `(recipe_id, day_of_week, meal_type)` 的唯一性**
3. 如果同一天生成了多个相同meal_type的菜品，后端会保存多个，但前端映射方式只显示最后一个

### 前端映射方式分析
前端使用 `organizedRecipes[day_of_week][meal_type] = detail` 的映射方式，这会直接覆盖相同meal_type的值。

### 修复目标
确保同一天、同一meal_type只能保存一道菜到数据库。

#### 修改后的代码
```python
# 保存当天菜品到数据库
saved = []
duplicate_found = False
mealtypes_for_day = set()

for detail_data in day_result["details"]:
    # 移除算法内部字段
    detail_data.pop("nutrition_score", None)
    detail_data.pop("tags", None)
    detail_data["day_of_week"] = day

    # 检查菜品名称是否重复（全局去重）
    existing_dish = db.query(RecipeDetail).filter(
        RecipeDetail.recipe_id == recipe_id,
        RecipeDetail.dish_name == detail_data["dish_name"]
    ).first()

    if existing_dish:
        logger.warning(f"[Step-by-Step Day {day}] 菜品名称已存在: {detail_data['dish_name']}，跳过")
        duplicate_found = True
        continue

    # 检查同一天同一餐次是否重复（关键修复）
    existing_mealtype = db.query(RecipeDetail).filter(
        RecipeDetail.recipe_id == recipe_id,
        RecipeDetail.day_of_week == day,
        RecipeDetail.meal_type == detail_data["meal_type"]
    ).first()

    if existing_mealtype:
        logger.warning(f"[Step-by-Step Day {day}] 餐次 {detail_data['meal_type']} 已存在菜品: {existing_mealtype.dish_name}，跳过重复菜品 {detail_data['dish_name']}")
        duplicate_found = True
        continue

    # 检查当天批次内的餐次是否重复（防止单批次内重复）
    if detail_data["meal_type"] in mealtypes_for_day:
        logger.warning(f"[Step-by-Step Day {day}] 单批次内餐次 {detail_data['meal_type']} 重复，跳过菜品 {detail_data['dish_name']}")
        duplicate_found = True
        continue

    detail = RecipeDetail(
        recipe_id=recipe_id,
        **detail_data
    )
    db.add(detail)
    db.flush()
    db.refresh(detail)

    # 包含ID用于前端显示
    saved.append({
        "id": detail.id,
        **detail_data
    })
    # 添加菜名到去重列表
    generated_dish_names.append(detail_data["dish_name"])
    mealtypes_for_day.add(detail_data["meal_type"])
```

## 修复说明

### 新增的去重逻辑

1. **全局菜品名称去重**（原有逻辑保留）
   - 检查整个recipe_id下是否存在相同的dish_name
   - 如果存在，跳过并记录警告

2. **同一天同餐次去重**（新增）
   - 检查同一天(day_of_week)、同一餐次(meal_type)是否已有菜品
   - 如果存在，跳过并记录警告，显示已存在的菜品名

3. **单批次内餐次去重**（新增）
   - 检查当前生成批次内是否已有该餐次
   - 如果存在，跳过并记录警告
   - 使用集合 `mealtypes_for_day` 跟踪当天已处理的餐次

### 日志改进

将原有的英文日志改为中文，更清晰地说明：
- 菜品名称重复
- 餐次重复（含已存在菜品名）
- 单批次内餐次重复

## 核心改动思路

1. **三层去重保护**：
   - 第一层：全局菜品名称去重（防止重复菜名）
   - 第二层：数据库查询同一天同餐次是否已存在（防止跨批次重复）
   - 第三层：当前批次内餐次去重（防止批次内重复）

2. **清晰的日志记录**：
   - 记录每种重复情况的具体信息
   - 餐次重复时显示已存在的菜品名
   - 帮助定位问题来源

3. **无侵入性修改**：
   - 不修改数据库结构
   - 不改变API接口
   - 仅在应用层增加去重逻辑

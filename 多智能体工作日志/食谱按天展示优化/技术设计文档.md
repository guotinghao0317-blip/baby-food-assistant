# 食谱"按天生成"技术设计文档

> 日期：2026-04-22
> 目标：将"逐菜流式生成"改为"按天生成"——一次生成当天3道菜，生成完后一次性展示当天全部食谱

---

## 一、现状分析

### 1.1 后端现状（`recipe_generator.py`）

`generate_next_day_stream` 函数当前按"逐菜"粒度推送 SSE 事件：

```
day_started → [dish_started → dish_stream(N次) → dish_done] x 3餐 → day_done
```

- `day_started`：通知前端开始生成第N天
- `dish_started`：通知前端开始生成某餐次（breakfast/lunch/dinner）
- `dish_stream`：推送单道菜的流式内容片段（打字机效果），包含 `content` 和 `full_content`
- `dish_done`：单道菜完成，携带完整 `detail` 数据
- `dish_failed`：单道菜生成失败
- `day_done`：当天3道菜全部完成，携带 `details` 数组

问题：
- 后端为每道菜做流式推送，逻辑复杂（每道菜要处理 CodingPlan 流式 / 算法降级 / 去重检查 / 数据库写入），代码量大且嵌套深
- `dish_stream` 事件的 `full_content` 字段在算法降级路径下模拟了3个阶段（菜名→食材→完整数据），是人为模拟的"流式效果"，并非真正的 LLM streaming
- 逐菜粒度的 SSE 事件迫使前端维护极复杂的 per-dish 状态机

### 1.2 前端现状（`[id]/page.tsx`）

当前前端的类型和状态管理极其复杂：

**类型定义：**
- `DishGenerationState`：每道菜有 5 种状态（pending/generating/streaming/done/failed），加上 `detail`、`streamingContent`
- `DayGenerationState`：每天有 3 种状态（pending/generating/done），同时维护 `details: DishData[]` 和 `dishes: Record<string, DishGenerationState>` 两套数据结构

**状态管理：**
- `streamingContentRef`：`Record<string, string>`，key = `"day-mealType"`，缓存流式内容
- `streamingContentMap`：`Record<string, string>`，带 200ms 节流的渲染用流式内容
- `streamingThrottleRef`：节流时间戳记录
- `handleSSEEvent`：处理 7 种事件类型（started/day_started/dish_started/dish_stream/dish_done/dish_failed/day_done/finished/error），每个事件都需要更新多层嵌套状态

**渲染逻辑：**
- `renderDishCard`：根据 DishGenerationState 的 5 种状态渲染不同 UI（pending占位/generating loading/streaming 打字机/done 完整/failed 失败）
- `getDisplayDishes`：从 `dishes` Record 提取显示列表，兼容 `details` 数组，兼容旧数据 snack
- `getCurrentDayDishes`：另一套提取逻辑，优先用 `dishes`，兼容 `details`

**核心问题：**
- 前端花了大量代码在"逐菜流式"的中间状态管理上（streaming 节流、缓存清理、dishes/details 双数据源同步）
- 多次迭代证明这种逐菜粒度的交互效果难以调到用户满意

### 1.3 其他前端页面

- `generate/page.tsx`：使用了相同的 DishGenerationState 体系和 dish 级事件处理，代码重复度高
- `generate-step-by-step/page.tsx`：同上，也是逐菜流式处理

## 二、目标方案设计

### 2.1 核心思路

**简化到极致：一天一个生命周期。**

- 后端：生成当天3道菜（内部仍是逐菜生成+保存），但只对外暴露 `day_started` → `day_done` 两个 SSE 事件
- 前端：当天在 loading 状态，3道菜全部完成后一次性展示，没有中间的 per-dish 状态

### 2.2 后端改造方案

#### 2.2.1 简化 SSE 事件流

**当前事件流（逐菜模式）：**
```
day_started
  dish_started(breakfast) → dish_stream x N → dish_done(breakfast)
  dish_started(lunch)     → dish_stream x N → dish_done(lunch)
  dish_started(dinner)    → dish_stream x N → dish_done(dinner)
day_done
```

**目标事件流（按天模式）：**
```
day_started(day=1)
day_done(day=1, details=[...3道菜完整数据])
day_started(day=2)
day_done(day=2, details=[...3道菜完整数据])
...
finished(recipe_id=xxx, total_dishes=21)
```

#### 2.2.2 `generate_next_day_stream` 函数改造

保留内部逐菜生成+保存的逻辑（这是业务需要：去重、数据库写入、降级），但**删除所有 `dish_started`/`dish_stream`/`dish_done`/`dish_failed` 事件的 yield**。

改造要点：
1. 函数开头 yield `day_started`
2. 内部 for 循环仍遍历 MEAL_TYPES，逐菜生成、去重检查、数据库写入——但不再 yield 中间事件
3. 遇到某道菜生成失败，记录到 `failed_meals` 列表，继续生成下一道（不中断）
4. 循环结束后，yield `day_done`，payload 包含：
   - `day`: 天号
   - `details`: 成功生成的菜品数据数组
   - `failed_meals`: 失败的餐次列表（如有）
   - `all_completed`: 是否7天全部完成

伪代码：
```python
async def generate_next_day_stream(...) -> AsyncGenerator[str, None]:
    yield day_started

    day_saved_details = []
    failed_meals = []

    for meal_type in MEAL_TYPES:
        dish_data = None
        # ... CodingPlan 优先 + 算法降级（保留原有逻辑，不 yield dish_stream）
        # ... 去重检查、数据库写入（保留原有逻辑）
        if success:
            day_saved_details.append(saved_detail)
        else:
            failed_meals.append(meal_type)

    # 检查是否7天完成
    all_completed = check_all_days_completed(recipe_id, db)

    yield day_done(day, details=day_saved_details, failed_meals=failed_meals, all_completed=all_completed)

    if all_completed:
        yield finished(recipe_id)
```

#### 2.2.3 `generate_weekly_recipe_step_by_step` 函数同步改造

同理，删除该函数内的 `dish_started`/`dish_stream`/`dish_done`/`dish_failed` 事件 yield，改为只推送 `day_started` → `day_done`。

#### 2.2.4 路由层 `recipes.py` 改造

`/generate-next-day/{recipe_id}` 端点无需改动（它只是调用 `generate_next_day_stream` 并包装成 StreamingResponse）。

`/generate-step-by-step` 端点同理，无需改动。

注释文档需更新，删除已废弃的事件类型说明。

### 2.3 前端改造方案

#### 2.3.1 删除的类型和状态

**删除：**
- `DishGenerationState` 接口（整个类型）
- `DayGenerationState.dishes` 字段
- `streamingContentRef` / `streamingContentMap` / `streamingThrottleRef` 三个 ref
- `STREAMING_MEAL_LABELS` 常量
- `getStreamingDisplayText` 函数
- `getCurrentDayDishes` 函数
- `getDisplayDishes` 函数

**简化后：**
```typescript
interface DishData {
  id: number
  meal_type: string
  dish_name: string
  ingredients: Array<{ name: string; amount: string }>
  cooking_steps: Array<{ step: number; description: string }>
  nutrition_info: { calories: number; protein: number; iron: number }
  image_url?: string
}

interface DayGenerationState {
  day: number
  status: 'pending' | 'generating' | 'done'
  details?: DishData[]  // 当天3道菜的完整数据
}
```

#### 2.3.2 `handleSSEEvent` 简化

只处理 4 种事件：

```typescript
const handleSSEEvent = (event: any) => {
  switch (event.type) {
    case 'day_started':
      // 将该天标记为 generating，自动选中
      setDays(prev => prev.map(d =>
        d.day === event.day ? { ...d, status: 'generating' } : d
      ))
      setSelectedDay(event.day)
      break

    case 'day_done':
      // 将该天标记为 done，填入菜品数据
      setDays(prev => prev.map(d =>
        d.day === event.day
          ? { ...d, status: 'done', details: event.details }
          : d
      ))
      break

    case 'finished':
      setOverallStatus('finished')
      break

    case 'error':
      setErrorMessage(event.message)
      setOverallStatus('error')
      break
  }
}
```

从 ~150 行降到 ~30 行。

#### 2.3.3 渲染逻辑简化

`renderDayContent`（替换原来的 `renderDishCard` + `getDisplayDishes`）：

```typescript
const renderDayContent = (dayState: DayGenerationState) => {
  if (dayState.status === 'pending') {
    return <等待生成占位>
  }
  if (dayState.status === 'generating') {
    return <当天loading动画，如"正在生成周X食谱...">
  }
  if (dayState.status === 'done' && dayState.details) {
    return <3道菜卡片一次性展示>
  }
}
```

没有 per-dish 状态，没有 streaming 节流，没有 dishes/details 双数据源。

#### 2.3.4 `generate/page.tsx` 同步改造

该页面同样删除 DishGenerationState、streaming 相关逻辑，改为按天渲染。

#### 2.3.5 `generate-step-by-step/page.tsx` 同步改造

同上。

## 四、验收标准

### 功能验收

1. **按天生成体验**：点击生成后，第1天显示 loading → 3道菜同时出现 → 第2天显示 loading → 3道菜同时出现 → ... → 第7天完成
2. **无逐菜中间状态**：前端不再出现"早餐生成中...午餐等待中..."这种 per-dish 状态，当天只有两种状态：loading / 完成
3. **自动切换**：正在生成的天自动被选中，生成完成后可切换查看其他已完成的天
4. **7天Tab状态**：pending(灰) / generating(pink loading) / done(绿 + 勾)

### 数据验收

5. **后端数据完整性**：每天3道菜（breakfast/lunch/dinner）全部写入数据库
6. **去重正常**：28道菜名称不重复
7. **降级正常**：CodingPlan API 不可用时自动降级到内置算法
8. **失败处理**：某道菜生成失败不影响其他菜品，day_done 的 failed_meals 字段记录失败餐次

### 代码质量验收

9. **前端代码量**：`[id]/page.tsx` 中 handleSSEEvent 从 ~150 行降至 ~30 行
10. **无死代码**：删除所有 DishGenerationState、streamingContentRef、streamingThrottleRef、STREAMING_MEAL_LABELS、getStreamingDisplayText 等不再使用的代码
11. **类型安全**：TypeScript 无编译错误

### 兼容性验收

12. **换一道菜功能**：不受本次改造影响，仍正常工作
13. **缓存兼容**：localStorage 缓存格式变更，旧缓存应能正常失效重载
14. **旧数据兼容**：已生成的食谱（status=completed）正常展示，不受新代码影响

## 五、风险与注意事项

1. **生成等待时间**：按天模式下，用户需要等待当天3道菜全部生成完才能看到内容（约10-15秒），期间只有一个 loading 动画。这是设计意图，但需要 loading 动画做得足够友好（如"正在为宝宝精心搭配周一食谱..."）
2. **部分失败展示**：如果当天3道菜中有1道失败，`day_done` 仍会推送成功的2道菜。前端需要处理 details 不足3道菜的情况
3. **缓存格式变更**：DayGenerationState 结构变化后，旧的 localStorage 缓存会加载失败。需要在 loadCache 中增加格式校验，不匹配时清除旧缓存
4. **三个前端页面同步改造**：`[id]/page.tsx`、`generate/page.tsx`、`generate-step-by-step/page.tsx` 三个页面有大量重复代码，建议改造时考虑抽取公共 hook（如 `useRecipeGeneration`），但这是后续优化项，本次不改

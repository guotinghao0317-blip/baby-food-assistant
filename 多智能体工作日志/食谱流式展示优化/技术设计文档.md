# 食谱流式展示优化 - 技术设计文档

## 1. 问题分析

### 1.1 问题一：流式生成策略问题

**现状**：当前系统采用"按天生成、按天返回"的策略。后端 `generate_weekly_recipe_step_by_step` 函数循环7天，每天一次性生成4道菜(早餐/午餐/晚餐/加餐)，通过SSE事件 `day_done` 一次性返回当天所有菜品。前端在收到 `day_done` 事件后才展示当天所有菜品。

**问题根因**：SSE事件粒度太粗——最小单位是"天"，即4道菜一起返回一起展示。用户希望粒度细到"每道菜"，即每道菜生成完毕就立即展示。

**涉及代码**：
- 后端：`backend/app/services/recipe_generator.py` 中 `generate_weekly_recipe_step_by_step()` 函数，L546-L693
- 后端：`backend/app/routers/recipes.py` 中 `/generate-stream` 和 `/generate-step-by-step` 路由
- 前端：`frontend/app/recipes/generate/page.tsx` 中 `day_done` 事件处理，L235-L259

### 1.2 问题二：页面不主动刷新问题

**现状**：存在两个生成页面：
1. `generate/page.tsx` — 流式生成页，生成完成后自动跳转到 `[id]/page.tsx`
2. `[id]/page.tsx` — 食谱详情页，该页面使用轮询模式(`generate-next-day`)获取数据

在详情页 `[id]/page.tsx` 中，`generateNextDay` 函数调用 `/generate-next-day/{recipe_id}` 接口，该接口返回的是**普通JSON响应**（非SSE流）。前端在 `day_done` 事件触发时才更新 `days` 状态，但 `selectedDay` 可能不指向正在生成的天，导致用户看到"等待生成中"但数据已经入库。

**问题根因**：
- 后端 `/generate-next-day/{recipe_id}` 返回的JSON中已包含完整的 `details`，但前端没有在收到响应后立即更新当前选中天的UI
- 前端 `handleSSEEvent` 中 `day_done` 只更新了对应天的 `days` 状态，但如果 `selectedDay !== event.day`，页面不会主动切换到新生成的天
- 更关键的是：前端虽然做了 `startDishEntranceAnimation`，但菜品是一次性全部出现的（`renderedDishes` 从0跳到总数），并非真正的"打字机效果"

**涉及代码**：
- 前端：`frontend/app/recipes/[id]/page.tsx`，L356-L413 (`generateNextDay`)，L416-L474 (`handleSSEEvent`)
- 前端：`frontend/app/recipes/[id]/page.tsx`，L476-L484 (`startDishEntranceAnimation`)

### 1.3 问题三：多道菜问题

**现状**：`recipe_algorithm.py` 中 `generate_day_meals` 函数每天生成4道菜（早餐、午餐、晚餐、加餐）。火山引擎API的prompt也要求"每天包含：早餐、午餐、晚餐、加餐（共4餐）"。

**问题根因**：用户只需要每餐一道菜，但当前系统每天每餐各生成一道，共4道。用户反馈的"多道菜"可能指两种理解：
- 理解A：每天只需要3道菜（早/午/晚），不要加餐
- 理解B：每餐只生成一道菜（当前已经如此），但展示时看起来太多

经代码分析确认：当前每餐确实只生成一道菜，但每天有4餐(含加餐)。用户反馈"多道菜"大概率是指**每天4餐太多，希望减为3餐（早/午/晚），去掉加餐**，或者用户觉得**一个页面展示4道菜信息量太大**。

**涉及代码**：
- 后端：`backend/app/services/recipe_algorithm.py`，L209-L298 (`generate_day_meals`)，`meal_types = ["breakfast", "lunch", "dinner", "snack"]`
- 后端：`backend/app/services/volcengine_client.py`，L258-L324 (`build_weekly_prompt`)，prompt中写死了4餐

## 2. 技术方案

### 2.1 核心思路：SSE事件粒度从"天"细化到"菜"

将SSE事件类型从：
```
day_started -> day_done(含当天所有菜品) -> ...
```
改为：
```
day_started -> dish_started -> dish_stream(打字机内容) -> dish_done(单道菜完成) -> ... -> day_done
```

这样前端可以在收到 `dish_done` 时立即渲染该菜品，不用等当天所有菜品生成完。

### 2.2 后端改造方案

#### 2.2.1 新增SSE事件类型

| 事件类型 | 说明 | 数据结构 |
|---------|------|---------|
| `started` | 生成开始 | `{type: "started", recipe_id: int}` |
| `day_started` | 某天开始生成 | `{type: "day_started", day: int}` |
| `dish_started` | 单道菜开始生成 | `{type: "dish_started", day: int, meal_type: string}` |
| `dish_stream` | 单道菜打字机内容流 | `{type: "dish_stream", day: int, meal_type: string, content: string, full_content: string}` |
| `dish_done` | 单道菜生成完成 | `{type: "dish_done", day: int, meal_type: string, detail: object}` |
| `day_done` | 某天所有菜品生成完成 | `{type: "day_done", day: int, details: array}` |
| `finished` | 全部7天完成 | `{type: "finished", recipe_id: int, total_dishes: int}` |
| `error` | 错误 | `{type: "error", message: string}` |

#### 2.2.2 后端生成逻辑改造

修改 `generate_weekly_recipe_step_by_step` 函数，将内层循环从"先批量生成再批量保存"改为"逐菜生成、逐菜保存、逐菜推送"：

**算法路径（降级方案）**：`generate_day_meals` 目前是一次性生成4道菜，改为循环调用新的 `generate_single_meal` 函数，每生成一道菜就保存并推送 `dish_done` 事件。

**CodingPlan路径（火山引擎API）**：当前是调用 `volcengine_client.generate_weekly_recipe` 一次性生成1天所有菜品。改为调用新的 `generate_single_meal_stream` 接口，每次只生成1道菜，并流式返回内容。

**关键改动**：
- 新增 `generate_single_meal(baby_profile, meal_type, exclude_names, day)` 函数
- 新增火山引擎prompt构建方法 `build_single_meal_prompt(baby, nutrition, meal_type, day, exclude_dish_names)`
- `generate_weekly_recipe_step_by_step` 内层循环改为：for meal_type in meal_types -> 生成单菜 -> 保存 -> yield dish_done

#### 2.2.3 去掉加餐（解决多道菜问题）

将 `meal_types` 从 `["breakfast", "lunch", "dinner", "snack"]` 改为 `["breakfast", "lunch", "dinner"]`。

**影响范围**：
- `recipe_algorithm.py` 的 `generate_day_meals` 函数
- `volcengine_client.py` 的 `build_weekly_prompt` 函数
- 前端的 `MEAL_TYPES` 常量
- 前端展示中所有引用 `snack` 餐次的地方

> 注意：这是一个用户可配置项，建议在Baby模型或配置中新增 `meals_per_day` 字段（默认3），以便未来灵活扩展。初版可硬编码为3餐。

### 2.3 前端改造方案

#### 2.3.1 核心状态改造

当前前端 `days` 状态结构：
```typescript
interface DayGenerationState {
  day: number
  status: 'pending' | 'generating' | 'done'
  details?: DishData[]  // 当天所有菜品，一次性赋值
}
```

改为：
```typescript
interface DishGenerationState {
  meal_type: string
  status: 'pending' | 'generating' | 'streaming' | 'done'
  detail?: DishData
  streamingContent?: string  // 打字机效果累积的内容
}

interface DayGenerationState {
  day: number
  status: 'pending' | 'generating' | 'done'
  dishes: Record<string, DishGenerationState>  // key = meal_type
}
```

#### 2.3.2 SSE事件处理改造

`handleSSEEvent` 新增处理：

```typescript
case 'dish_started':
  // 标记该菜品开始生成，显示loading状态
  setDays(prev => prev.map(d =>
    d.day === event.day
      ? { ...d, dishes: { ...d.dishes, [event.meal_type]: { meal_type: event.meal_type, status: 'generating' } } }
      : d
  ))
  break

case 'dish_stream':
  // 打字机效果：逐步显示内容
  setDays(prev => prev.map(d =>
    d.day === event.day
      ? { ...d, dishes: { ...d.dishes, [event.meal_type]: { ...d.dishes[event.meal_type], status: 'streaming', streamingContent: event.full_content } } }
      : d
  ))
  break

case 'dish_done':
  // 单道菜完成，立即展示
  setDays(prev => prev.map(d =>
    d.day === event.day
      ? { ...d, dishes: { ...d.dishes, [event.meal_type]: { meal_type: event.meal_type, status: 'done', detail: event.detail } } }
      : d
  ))
  // 自动选中正在生成的天
  setSelectedDay(event.day)
  break
```

#### 2.3.3 渲染逻辑改造

当前菜品渲染逻辑：`currentDetails.map()` 遍历所有菜品，配合 `renderedDishes` 控制逐步显示。

改为：遍历 `days[selectedDay].dishes`，根据每个菜品的 `status` 分别渲染：
- `pending`：显示"等待生成"占位卡片
- `generating`：显示loading动画
- `streaming`：显示打字机效果内容（解析 `streamingContent` 渲染为结构化数据）
- `done`：显示完整菜品信息

#### 2.3.4 页面主动刷新

在 `dish_done` 事件处理中，自动将 `selectedDay` 切换到正在生成的天，确保用户立即看到新菜品。

同时，在 `day_done` 事件中，如果当前没有选中任何生成中的天，自动切换到下一个正在生成的天。

### 2.4 流式输出中的打字机效果实现

#### 方案说明

后端通过 `dish_stream` 事件推送LLM的原始流式输出（文本片段），前端累积后解析为结构化数据并逐步渲染。

对于**火山引擎API路径**：直接使用 `stream_generate` 获取LLM流式输出，每次收到chunk就推送 `dish_stream` 事件。LLM输出完成后解析JSON，推送 `dish_done` 事件。

对于**算法降级路径**：由于数据是即时生成的（不需要等待LLM），可以模拟打字机效果——将已生成的结构化数据逐步推送。具体实现：每道菜生成后，先将菜名推送（dish_stream），然后逐步推送食材、步骤等。

## 4. 接口定义

### 4.1 修改的API：`POST /api/recipes/generate-stream`

**变更说明**：SSE事件类型扩展，新增 `dish_started`、`dish_stream`、`dish_done` 事件。

**SSE事件序列（新版）**：
```
data: {"type": "started", "recipe_id": 123}

data: {"type": "day_started", "day": 1}

data: {"type": "dish_started", "day": 1, "meal_type": "breakfast"}

data: {"type": "dish_stream", "day": 1, "meal_type": "breakfast", "content": "胡萝卜", "full_content": "胡萝卜"}

data: {"type": "dish_stream", "day": 1, "meal_type": "breakfast", "content": "泥", "full_content": "胡萝卜泥"}

... (更多stream事件)

data: {"type": "dish_done", "day": 1, "meal_type": "breakfast", "detail": {"id": 1, "day_of_week": 1, "meal_type": "breakfast", "dish_name": "胡萝卜泥", "ingredients": [...], "cooking_steps": [...], "nutrition_info": {...}}}

data: {"type": "dish_started", "day": 1, "meal_type": "lunch"}

... (更多dish事件)

data: {"type": "dish_done", "day": 1, "meal_type": "dinner", "detail": {...}}

data: {"type": "day_done", "day": 1, "details": [...]}

... (Day 2-7 类似)

data: {"type": "finished", "recipe_id": 123, "total_dishes": 21}
```

### 4.2 修改的API：`POST /api/recipes/generate-step-by-step`

同上，SSE事件类型扩展。

### 4.3 修改的API：`GET /api/recipes/generate-next-day/{recipe_id}`

**变更说明**：返回数据中去掉snack类型菜品。返回格式不变。

### 4.4 新增内部函数（非API）

#### `generate_single_meal(baby_profile, meal_type, exclude_names, day) -> Dict`

生成单道菜品的算法函数。

**参数**：
- `baby_profile: BabyProfile` - 宝宝信息
- `meal_type: str` - 餐次类型 (breakfast/lunch/dinner)
- `exclude_names: Set[str]` - 需要排除的菜品名称集合
- `day: int` - 天数 (1-7)

**返回**：
```python
{
    "day_of_week": day,
    "meal_type": meal_type,
    "dish_name": "胡萝卜泥",
    "ingredients": [{"name": "胡萝卜", "amount": "50g"}],
    "cooking_steps": [{"step": 1, "description": "胡萝卜洗净去皮"}],
    "nutrition_info": {"calories": 25, "protein": 0.5, "iron": 0.3}
}
```

#### `generate_single_meal_stream(baby, nutrition, meal_type, day, exclude_dish_names) -> AsyncGenerator[str, None]`

流式生成单道菜品的火山引擎API调用。

**SSE事件输出**：
- `dish_stream`：LLM流式内容
- 完成：解析JSON后交由调用方发送 `dish_done`

## 6. 验收标准

### 6.1 流式生成策略（问题一）

- [ ] 后端生成食谱时，SSE事件粒度为"单道菜"，不再按天批量推送
- [ ] 前端在收到 `dish_done` 事件后，该菜品立即出现在页面中，无需等待当天其他菜品
- [ ] 每道菜从开始生成到展示完成，用户可感知到逐菜出现的渐进体验
- [ ] 如果火山引擎API可用，使用 `dish_stream` 事件实现LLM输出的真实打字机效果
- [ ] 如果降级到内置算法，每道菜也有逐步呈现的效果（菜名先出现，然后食材，然后步骤）

### 6.2 页面主动刷新（问题二）

- [ ] 食谱生成过程中，正在生成的天自动被选中（`selectedDay` 自动切换）
- [ ] 每道菜生成完成后，页面内容自动更新，无需用户手动切换Tab
- [ ] 用户手动切换Tab后，如果新菜品生成完成，切回来时能看到最新数据
- [ ] 不存在"需要切换Tab才能看到数据"的情况

### 6.3 多道菜问题（问题三）

- [ ] 每天每餐只生成一道菜（早/午/晚共3道，不再有加餐）
- [ ] 前端 `MEAL_TYPES` 中不包含 "snack"/"加餐"
- [ ] 后端prompt中不要求生成加餐
- [ ] 数据库中已有包含snack的旧食谱不受影响（向下兼容）

### 6.4 通用质量标准

- [ ] 所有现有API保持向下兼容（旧的 `day_done` 事件格式仍然有效）
- [ ] 换一道菜功能（`replace-dish-stream`）不受影响
- [ ] 生成失败时正确推送 `error` 事件，前端显示错误提示
- [ ] 页面在移动端和桌面端均正常显示
- [ ] 中断生成后重新进入页面，能从已生成的天数继续
- [ ] 流式输出期间，页面不出现闪烁或跳动

## 7. 风险与注意事项

1. **火山引擎API响应延迟**：逐菜生成意味着LLM调用次数从1次(1天)变为3次(3道菜)，总耗时会增加。建议在前端显示"正在生成第X道菜"的进度提示。

2. **算法降级路径的打字机效果**：内置算法是即时生成的，没有真实的流式内容。模拟打字机效果需要额外设计（如逐步推送结构化字段），实现复杂度较高。建议初版先用"生成完即展示"的方式，后续迭代再添加模拟打字机效果。

3. **SSE事件顺序保证**：需要确保 `dish_started` -> `dish_stream` -> `dish_done` 的顺序不被乱序。由于是单线程async生成，天然有序。

4. **前端状态更新频率**：`dish_stream` 事件可能非常频繁（每秒多个），需要注意React状态更新性能。建议使用 `useRef` 缓存流式内容，只在关键节点触发 `setState`。

5. **去掉加餐的向下兼容**：已有食谱中可能包含snack类型数据，前端展示时需要兼容（如果数据库中有snack数据就显示，没有就不显示），不能硬报错。

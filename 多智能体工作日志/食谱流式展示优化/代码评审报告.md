### 阻断性问题 (P0 - 必须修复)

#### 1. `generate-next-day` 端点返回JSON但前端按SSE解析 — 两个页面完全无法接收dish级事件

**位置**: `backend/app/routers/recipes.py:449-533` + `frontend/app/recipes/[id]/page.tsx:388-443` + `frontend/app/recipes/generate-step-by-step/page.tsx:362-418`

**问题描述**:
后端 `GET /api/recipes/generate-next-day/{recipe_id}` 路由使用 `generate_next_day_sync` 函数，返回普通JSON对象：
```python
# recipes.py:522-533
result = await generate_next_day_sync(...)
result["recipe_id"] = recipe_id
return result  # 返回普通JSON
```

但前端两个页面(`[id]/page.tsx` 和 `generate-step-by-step/page.tsx`)都用 `response.body?.getReader()` 读取流，并通过 `parseSSE` 解析SSE事件：
```typescript
// [id]/page.tsx:405-431
const reader = response.body?.getReader()
// ...
buffer += decoder.decode(value, { stream: true })
const events = buffer.split('\n\n')
for (const event of events) {
  const data = parseSSE(event)  // 期望 "data: " 前缀，JSON响应中没有
  if (!data) continue  // 永远为null，所有事件被跳过
  handleSSEEvent(data)  // 永远不会执行
}
```

**影响**:
- `[id]/page.tsx`（食谱详情页）：生成期间用户看不到任何进度，`dish_started`/`dish_stream`/`dish_done`/`day_done` 事件全部丢失。直到全部7天生成完毕后 `fetchRecipe()` 才刷新数据，用户体验等于退回到改动前的状态。
- `generate-step-by-step/page.tsx`（分步生成页）：同理，所有dish级事件丢失。
- 本次改动的核心目标——"逐菜流式展示"——在这两个页面上完全失效。
- `generate/page.tsx` 不受影响，因为其使用 `POST /api/recipes/generate-stream`（SSE端点）。

**改进建议**:
方案A（推荐）：将 `/generate-next-day` 路由改回SSE端点，使用已有的 `generate_next_day_stream` 函数（目前是死代码，已被import但未使用），并将其改造为输出dish级事件（`dish_started`/`dish_stream`/`dish_done`/`day_done`）。

方案B：修改前端 `generateNextDay` 函数，直接解析JSON响应，然后手动触发 `handleSSEEvent`。

#### 2. `parseStreamingContent` 在流式传输期间永远失败 — 打字机效果名存实亡

**位置**: `frontend/app/recipes/[id]/page.tsx:71-88` + `generate/page.tsx:67-81` + `generate-step-by-step/page.tsx:50-64`

**问题描述**:
`parseStreamingContent` 尝试 `JSON.parse(content)` 来解析流式内容为结构化数据。但在流式传输过程中，LLM输出是不完整的JSON（如 `{"detail": {"dish_name": "胡萝卜`），`JSON.parse` 必然抛出异常，函数返回 `null`。只有当整个JSON完整时才能解析成功，此时 `dish_done` 事件也已到达。

```typescript
const parseStreamingContent = (content: string) => {
  try {
    const parsed = JSON.parse(content)  // 流式期间永远失败
    return parsed
  } catch {
    return null  // 永远走这个分支
  }
}
```

**影响**:
- `streaming` 状态的渲染总是走 fallback 分支，显示原始文本（可能是部分JSON如 `{"detail": {"dish_name": "胡萝卜`），用户体验极差。
- 技术设计文档FE-4要求"在 dish_stream 事件中累积内容，解析后以打字机效果展示菜名、食材、步骤"，但当前实现无法达成。
- `animate-blink` 光标动画配合原始JSON文本显示，效果反直觉。

**改进建议**:
方案A（推荐）：实现增量JSON解析器，在流式内容中用正则提取已完整的字段（如当检测到 `"dish_name": "xxx"` 完整闭合时提取菜名）。

方案B（快速实现）：在流式期间显示简洁的状态文案（如"正在生成早餐..."），不做结构化解析。等 `dish_done` 事件到达后再展示完整菜品。技术设计文档7.2也提到"初版先用'生成完即展示'的方式"。

### 严重问题 (P1 - 应尽快修复)

#### 3. `dish_done` 事件中 `detail: None` 仍将菜品标记为 `done` — 数据不一致

**位置**: `backend/app/services/recipe_generator.py:650,656,673,685` + 前端各页面 `dish_done` 处理逻辑

**问题描述**:
当菜品生成失败（算法返回None、数据无效、重复等）时，后端发送：
```python
yield f"data: {json.dumps({'type': 'dish_done', 'day': day, 'meal_type': meal_type, 'detail': None, ...})}\n\n"
```

前端 `dish_done` 处理逻辑无条件将状态设为 `done`：
```typescript
[event.meal_type]: {
  meal_type: event.meal_type,
  status: 'done' as const,  // 即使detail为null也标记为done
  detail: event.detail       // 此处为null
}
```

随后的 `allDone` 检查：
```typescript
const allDone = ACTIVE_MEAL_TYPES.every(mt =>
  newDishes[mt]?.status === 'done'  // status='done'即使detail=null
)
```

**影响**:
- 菜品生成失败时，该天的状态仍可能被标记为 `done`，导致天卡片显示绿色已完成标识。
- 前端 `renderDishCard` 中 `status === 'done' && dishState.detail` 条件会跳过渲染，该菜品卡片消失。
- 天状态为 `done` 但菜品数不足3道，数据不一致。

**改进建议**:
- 后端：生成失败时发送新事件类型 `dish_failed`，或 `dish_done` 中增加 `success: false` 标记。
- 前端：`dish_done` 处理时检查 `event.detail` 是否为 null/undefined，若是则将状态设为 `failed` 而非 `done`。

#### 4. `dish_stream` 事件无节流 — 高频setState导致性能隐患

**位置**: `frontend/app/recipes/[id]/page.tsx:501-527` + `generate/page.tsx:240-262` + `generate-step-by-step/page.tsx:187-211`

**问题描述**:
技术设计文档7.4明确指出："dish_stream 事件可能非常频繁（每秒多个），建议使用 useRef 缓存流式内容，只在关键节点触发 setState"。

代码注释也写了"使用 ref 缓存流式内容，减少 setState 频率"，但实际实现中**每次 `dish_stream` 事件都调用 `setDays`**，`streamingContentRef` 只用于缓存清理，从未用于节流：

```typescript
// [id]/page.tsx:501-527
case 'dish_stream': {
  const cacheKey = `${event.day}-${event.meal_type}`
  const newContent = event.full_content || ''
  streamingContentRef.current[cacheKey] = newContent  // 写入ref

  // 但紧接着又立即调用 setDays，ref没有起到节流作用
  setDays(prev => prev.map(d => { ... }))
  break
}
```

**影响**:
- 火山引擎API流式输出时，token级别的事件可达每秒10+次，每次触发 `setDays` 导致整个 `days` 数组重建和重新渲染。
- 移动端设备可能出现UI卡顿。
- 由于问题#2，`parseStreamingContent` 在streaming期间总是返回null，导致每次setState都触发无意义的渲染（显示原始JSON文本）。

**改进建议**:
使用 `useRef` + `requestAnimationFrame` 或时间间隔节流实现真正的节流：
```typescript
const lastUpdateTime = useRef<number>(0)
const pendingUpdate = useRef<any>(null)

case 'dish_stream': {
  const cacheKey = `${event.day}-${event.meal_type}`
  streamingContentRef.current[cacheKey] = event.full_content || ''

  const now = Date.now()
  if (now - lastUpdateTime.current > 100) {  // 每100ms最多更新一次UI
    lastUpdateTime.current = now
    setDays(/* ... */)
  }
  break
}
```

#### 14. `db.commit()` 在逐菜生成循环中每道菜提交一次

**位置**: `backend/app/services/recipe_generator.py:696`

**问题描述**:
```python
db.add(detail)
db.flush()
db.refresh(detail)
db.commit()  # 每道菜都提交
```

逐菜commit确保数据持久化（即使中途失败已生成的菜品不会丢失），但增加了I/O开销。7天 x 3餐 = 21次commit。

**改进建议**:

## 总体评价

### 优点

1. **设计思路正确**：SSE事件粒度从"天"细化到"菜"的方案设计合理，能够显著改善用户感知等待时间。
2. **向后兼容处理到位**：前端对旧数据中snack的兼容处理（`getDisplayDishes` 中双路径检查）非常完善，不会因去掉加餐导致旧数据展示异常。
3. **`day_done` 旧版事件兼容**：前端处理 `day_done` 时既支持新版（有dishes Record）也支持旧版（有details数组），保证过渡期不报错。
4. **后端双路径降级机制完善**：CodingPlan优先 + 算法降级的策略在逐菜生成中得到了正确延续。
5. **算法层 `generate_single_meal` 设计良好**：复用 `filter_recipes` 和 `select_diverse_meal`，避免了代码重复。
6. **CSS动画实现**：`animate-blink` 打字机光标效果简洁有效。

### 核心风险

本次改动存在 **2个阻断性Bug**：

1. **`generate-next-day` 端点返回JSON但前端按SSE解析**：导致 `[id]/page.tsx` 和 `generate-step-by-step/page.tsx` 两个核心页面在生成期间完全无法显示进度，等于流式展示优化在这两个页面上没有生效。这是本次变更最大的功能回归。

2. **`parseStreamingContent` 在流式期间永远解析失败**：即使问题#1修复后，`generate/page.tsx` 上的streaming状态也只能显示原始JSON文本而非结构化内容，打字机效果名存实亡。

### 修复优先级建议

| 优先级 | 问题编号 | 修复工作量 |
